## <span style="color:#4EA1FF;">Обзор</span>

**MatrixClock** — это актуальная рабочая прошивка часов для матрицы `32x8`, собранная вокруг `ESP32-S3`.

Текущий фокус проекта:
- аккуратный рендер часов на компактной LED-матрице
- настройка через браузер
- OTA-обновление прошивки
- NTP-синхронизация времени
- дополнительные экраны с датчиками
- спектрум-режимы
- один финальный режим огня: `Fire`, работающий от файловой анимации в `LittleFS`
- проект сознательно не расширяется новыми пользовательскими функциями, пока не будет закрыт `Stage 1`

Этот репозиторий отражает **текущее состояние проекта**, а не архив старых экспериментов. Старые процедурные ветки огня и промежуточный генерационный мусор были убраны в пользу текущей runtime-схемы.

Текущая контрольная точка:
- фундамент `Stage 1` уже в основном собран
- но продуктовый Wi-Fi onboarding, переносимый OTA path и остаточные кодировочные хвосты ещё не считаются окончательно закрытыми, хотя cleanup runtime-entrypoints уже начат

## <span style="color:#4EA1FF;">Контур проекта</span>

В проекте теперь есть явный контур согласованности:

- перед правкой файлов сначала читается документация из `project/`

- общие правила этого контура описаны в [`PROJECT_EDIT_RULES.md`](./PROJECT_EDIT_RULES.md)
- async lifecycle рассматривается как важная зона риска и описывается по подсистемам, а не через автоматическое навязывание одной глобальной state machine
- если важная политика или механизм добавлены в `PLAN` / `RULES`, их короткое пояснение тоже должно появиться в `PROJECT_GUIDE_RU.md`

- shell-команды должны использоваться через стабильные повторяемые шаблоны, чтобы сохранённые разрешения не расползались по множеству почти одинаковых вариантов
- кодовые сущности и технические идентификаторы не переводятся; обязательная полная русификация относится прежде всего к русскому UI и русскоязычному объяснительному слою
- русские комментарии нельзя считать испорченными только по выводу PowerShell/терминала; для них источником истины считается сам файл в редакторе, а не консольное отображение
- экономия ресурса считается правилом проекта, но только там, где она не ухудшает надёжность, проверку и качество результата
- внешний обычный ChatGPT допускается как дешёвый draft-worker для узких локальных задач, но финальные решения, интеграция, проверка и документальная синхронизация остаются за Codex
- для внешнего GitHub-пакета нужно различать локальные пути `docs/...` и удалённые опубликованные пути `project/...` / `chatgpt_git/...`; эти две схемы не должны смешиваться в одном наборе ссылок

## <span style="color:#67C587;">Текущие возможности</span>

### <span style="color:#4EA1FF;">Часы и UI</span>
- шрифты `3x5`, `4x7`, `5x7`
- режимы анимации двоеточия
- режимы progress bar
- web UI из `LittleFS`
- web UI перестроен на вкладки для более короткой и управляемой навигации
- текущая навигация Web UI: `Настройки`, `Интерфейс`, `Погода`, `Инфо`; timezone живёт в `Настройках`, яркость и внешний вид — в `Интерфейсе`
- сохранение настроек через `Preferences`
- ручной выбор timezone
- auto-detect timezone через браузер и IP path

### <span style="color:#FF9B5E;">Визуальные режимы</span>
- spectrum bars
- waterfall
- symmetric spectrum
- VU meter
- `Fire` на базе файловой анимации: `data/anim/fire_32x8_v1.anim`
- audio async теперь имеет явный runtime-status (`idle` / `starting` / `running` / `stopping` / `error`) вместо неявного старта FFT-задачи

### <span style="color:#67C587;">Сеть и время</span>
- базовая идея: часы должны оставаться часами без интернета, а сеть должна расширять устройство сервисами
- подключение к Wi-Fi
- backend-основа для AP onboarding: AP fallback, Wi-Fi scan и browser-side config endpoints
- browser-side Wi-Fi onboarding panel в Web UI: status, scan, SSID/password, save, save+reboot
- Wi-Fi onboarding теперь валидирует базовую корректность SSID/password и не уходит в reboot при явно плохой конфигурации
- Wi-Fi onboarding в AP-режиме теперь умеет планировать живую попытку перехода в пользовательскую сеть без обязательного reboot: конфиг сохраняется, после ответа Web UI запускается apply-path, а при неудаче setup AP остаётся активной
- Wi-Fi onboarding теперь отдаёт явный `applyState` (`scheduled` / `connecting` / `succeeded` / `failed`), а Web UI показывает живой статус попытки подключения вместо немого ожидания
- Wi-Fi scan работает через async backend + короткий cache, чтобы setup AP не подвисал на тяжёлом синхронном scan и не запускал его повторно от каждого клиента
- Wi-Fi credentials хранятся в `Preferences` / `NVS`, поэтому OTA-прошивка и обновление `LittleFS` не должны их сбрасывать
- добавлен физический recovery path: при удержании кнопки на `GPIO21` во время старта сохранённые Wi-Fi credentials очищаются, и часы сразу поднимают setup AP для новой настройки сети
- после recovery-кнопки устройство запоминает forced setup AP state: следующий reset не должен возвращать dev bootstrap сеть, пока пользователь не введёт новые Wi-Fi credentials
- в setup AP режиме матрица показывает подсказку `AP:` и адрес `192.168.4.1`, чтобы пользователь мог открыть настройку без бумажной инструкции
- файловая система теперь имеет явный runtime-status (`mounted` / `missing_content` / `unmounted`) вместо неявного поведения
- OTA теперь имеет явный runtime-status и отложенный reboot после успешного обновления, чтобы сетевой обмен успевал завершиться корректно
- Wi-Fi теперь тоже имеет явный status message, а OTA корректно переинициализируется после разрыва и возврата соединения
- shared `platformio.ini` теперь держит только общий OTA fallback (`matrixclock.local`), а machine-specific `upload_port` и `host_ip` вынесены в автоматически подхватываемый `platformio.local.ini`
- `platformio.local.ini` исключён из git и остаётся рабочим локальным слоем для кнопки `Upload`; это уже лучше, чем жёстко держать IP в общем конфиге, но финальным переносимым OTA path всё ещё не считается
- NTP-синхронизация
- optional `DS3231` RTC fallback: если модуль найден, системное время поднимается с RTC до прихода NTP
- после появления валидного system/NTP time часы синхронизируют DS3231 обратно, чтобы следующий холодный старт был автономным
- `AT24C32` на модуле RTC пока только детектируется и не используется как рабочее хранилище
- будущий интернет-смысл проекта: радио, погода и уведомления, но не зависимость базового времени от сети
- web-интерфейс
- OTA-обновление прошивки
- OTA-обновление файловой системы
- нормализация timezone и её сохранение в `Preferences`
- минимальные runtime-метрики через backend: `frameTimeUs`, `frameTimeAvgUs`, `freeHeapBytes`
- display/runtime contour теперь тоже имеет явный backend-status: `waiting_for_time`, `ip_overlay`, `clock_fade_in`, `running_clock`, `running_spectrum`

### <span style="color:#D98BFF;">Датчики</span>

## <span style="color:#FF9B5E;">Железо</span>

### <span style="color:#4EA1FF;">Целевой модуль</span>
- `ESP32-S3`
- ожидаемый класс модуля: `N16R8`
- flash: `16MB`
- PSRAM: `8MB`

### <span style="color:#4EA1FF;">Матрица</span>
- тип: `WS2812`
- размер: `32x8`
- data pin: `GPIO12`
- модель рендера: буферизованный вывод через `NeoPixelBus`

### <span style="color:#4EA1FF;">Аудиовход</span>
- микрофон подключается по I2S
- текущие I2S-пины заданы в `src/audio/spectrum_async.cpp`

### <span style="color:#4EA1FF;">Датчики</span>
- сенсорный блок работает через I2C
- текущая реализация поддерживает `AHT20` и `BMP280`

### <span style="color:#4EA1FF;">RTC модуль</span>
- optional модуль: `DS3231` на I2C
- адрес RTC: `0x68`
- адрес EEPROM на типовом модуле `ZS-042`: `0x57`
- RTC хранит время как UTC, а отображение локального времени остаётся задачей timezone/NTP слоя
- если DS3231 не найден, прошивка работает как раньше без отдельной сборки
- инженерная роль DS3231: автономный старт времени без интернета, а не отдельный пользовательский режим
- источник точной коррекции остаётся NTP; RTC нужен как backup между синхронизациями
- `AT24C32` не используется для будильников или настроек, пока нет отдельного обоснованного сценария

В setup AP режиме интерфейс должен оставаться автономным:

- точка настройки рассчитана на одного клиента, чтобы телефон/ПК не конкурировали между собой за слабый AP-канал ESP32;
- AP-страница должна быть понятной и работоспособной без интернета.

## <span style="color:#FF9B5E;">Режим огня</span>

В проекте теперь остался **один финальный режим огня**:

- имя режима: `Fire`
- источник: файловая анимация в `LittleFS`
- runtime-ассет: `data/anim/fire_32x8_v1.anim`
- визуальный source reference: `png/fire_32x8_v1.gif`
- runtime playback: `src/effects/fire_animation.*`
- файл `.anim` загружается один раз в RAM / PSRAM, дальше кадры проигрываются из памяти

Почему победил именно этот вариант:
- он выглядит лучше старого процедурного огня
- у него предсказуемое runtime-поведение
- он не раздувает код гигантскими animation header'ами
- он лучше ложится на текущую архитектуру проекта

Это и есть канонический путь огня в текущем MatrixClock.

## <span style="color:#67C587;">Примечания</span>

### <span style="color:#4EA1FF;">Wi-Fi</span>
Текущий режим разработки допускает жёсткий dev bootstrap SSID/password, чтобы часы сразу поднимались в рабочей сети разработчика.
При этом продуктовая логика уже отделяется от dev-режима: сохранённые credentials хранятся в persisted state и будут использоваться как основной путь для пользовательской сети.
Финальная цель для этого контура: часы поднимают AP, сканируют доступные Wi-Fi сети, позволяют через браузер выбрать сеть, ввести пароль и после reboot подключаются уже к сети пользователя.
Если устройство уже привязано к чужой или недоступной сети, recovery-кнопка на старте очищает Wi-Fi конфиг и возвращает часы в setup AP без необходимости USB-перепрошивки.

## <span style="color:#D98BFF;">Дорожная карта</span>

Текущий план стабилизации и рефакторинга проекта лежит в:

- [`REFACTOR_PLAN_RU.md`](./REFACTOR_PLAN_RU.md)

Там описаны исходные стадии, их живой статус, журнал изменений и изменения ценности задач по мере развития проекта.
Дополнительно в плане уже зафиксированы долгосрочные архитектурные направления:
- более явный frame pipeline
- возможная унификация эффектов
- развитие `Fire` поверх уже существующего RAM / PSRAM preload
- более глубокие runtime-метрики для контроля кадра
- интернет-сервисы как отдельный слой после стабилизации основы: радио, погода и уведомления

Возможные онлайн-уведомления считаются отдельным будущим контуром. Прямые интеграции с почтой, OLX, Etsy, TikTok, YouTube и похожими сервисами технически возможны только при аккуратной схеме авторизации и чаще всего требуют промежуточного сервера, телефона или внешнего bridge-сервиса. ESP32 не должен становиться местом хранения сложных OAuth-секретов и тяжёлой логики API, если это можно вынести наружу.

Первым лёгким интернет-сервисом выбран погодный контур. Предпочтительный стартовый источник — `Open-Meteo`, потому что он отдаёт простой JSON без обязательного API-ключа и без OAuth. Погода внедряется как отдельный `weather_service.*`, а не как логика внутри Web UI или clock runtime. Текущий backend-контур: пользовательская локация хранится в NVS, город ищется из браузера через `Open-Meteo Geocoding API`, координаты можно также ввести вручную или взять из браузера, HTTP-запрос погоды выполняется в отдельной FreeRTOS task, последний успешный ответ кэшируется, status/error отдаются через `/state`, а render/network loop не блокируется при проблемах сети. В Web UI погода отображается отдельной вкладкой `Погода`: имя локации/координаты, статус, событие, температура, влажность, давление, ветер, облачность и возраст обновления. Основной бытовой сценарий настройки — ввести город, выбрать правильный результат поиска и сохранить. Ручная широта/долгота остаётся запасным инженерным способом, если поиск не подходит.

В Web UI добавлена вкладка `Инфо`. Это не архитектурный источник истины, а встроенная пользовательская шпаргалка для бытовых сценариев: первичная настройка Wi-Fi/AP, настройка погоды, автономное время через RTC, обновления прошивки/файловой системы и проверка состояния через `/state`. Тексты в этой вкладке считаются редактируемой пользовательской прозой.

### <span style="color:#4EA1FF;">Идеи внешних модулей</span>

По важности для проекта:
- `must for stability`: нормальное питание матрицы, запас по току, общий GND, крупный конденсатор на 5V-линии и при необходимости level shifter `74AHCT125` / `74HCT245` для WS2812
- `should for radio`: `MAX98357A` как простой I2S-усилитель для будущего интернет-радио
- `optional for quality`: `PCM5102A`, если нужен линейный аудиовыход на внешний усилитель вместо встроенного динамика
- `later if needed`: `TCA9548A` для I2C, если появятся конфликтующие адреса, и `FRAM`, если когда-нибудь понадобится частая запись маленьких данных
- `avoid until justified`: microSD, дополнительная EEPROM-логика и второй полноценный UI-контур без конкретной причины

Сервисный дисплей `16x2` или `320x240 TFT` возможен, но не считается текущим приоритетом. Бытовая польза понятна: локальная диагностика и настройка без браузера. Инженерный риск тоже понятен: это второй UI, который нужно синхронизировать с Web UI, состоянием и настройками. Если такой экран появится, сначала он должен быть диагностическим (`IP`, Wi-Fi mode, RTC, FS/OTA status), а не полноценным дублем всех настроек.

Эти идеи не считаются срочными до закрытия `Stage 1`, но уже встроены в roadmap как будущие инженерные слои.

- GitHub-ссылки на проектные документы уже встроены в эти два файла, чтобы внешний ChatGPT мог опираться на удалённые правила, если умеет читать ссылки

- [`docs/chatgpt_git/CHATGPT_PROJECT_SETTINGS_RU.md`](../chatgpt_git/CHATGPT_PROJECT_SETTINGS_RU.md)
