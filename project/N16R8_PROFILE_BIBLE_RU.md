# <span style="color:#4EA1FF;">N16R8 Board Profile Bible</span>

> <span style="color:#AEBFD7;">Техническая памятка по тому, как правильно завести `ESP32-S3 N16R8` в PlatformIO, почему стандартный профиль может врать про `8MB`, и какие настройки обязательны для полноценной работы `16MB flash + 8MB PSRAM`.</span>

---

## <span style="color:#4EA1FF;">Что это за модуль</span>

Маркировка `N16R8` для `ESP32-S3` обычно означает:

- `N16` = `16MB Flash`
- `R8` = `8MB PSRAM`

В нашем проекте это подтвердилось не только надписью на модуле, но и реальным чтением параметров с чипа.

---

## <span style="color:#FF9B5E;">С какими симптомами мы столкнулись</span>

На практике проблема выглядела так:

- модуль был промаркирован как `N16R8`
- проект собирался
- обычная прошивка могла заливаться
- но `PlatformIO` / `uploadfs` вели себя так, как будто у платы только `8MB flash`
- файловая система и разметка упирались в `8388608 bytes`
- стандартный профиль `esp32-s3-devkitc-1` не давал честной картины по флешу

Из-за этого создавалось ощущение, что:

- либо железо урезано
- либо маркировка липовая
- либо VS Code / PlatformIO “видят” только половину памяти

---

## <span style="color:#4EA1FF;">Что было реальной причиной</span>

Корень проблемы был не в железе, а в профиле платы.

Стандартный профиль `esp32-s3-devkitc-1` в нашей связке по факту вёл проект как `N8`-вариант:

- часть параметров жила как у `8MB`-платы
- `uploadfs` и связанная логика ориентировались не на реальный модуль, а на board profile
- из-за этого `LittleFS` и partition-таблица не использовали всю флеш-память корректно

Иными словами:

- чип реально умел `16MB`
- но проект был описан как более узкая плата
- PlatformIO честно следовал профилю, а не наклейке на модуле

---

## <span style="color:#67C587;">Что подтвердило, что флеш реально 16MB</span>

Ключевым был не IDE, а чтение параметров напрямую с устройства.

Когда мы проверили чип инструментами низкого уровня, получили:

- `ESP32-S3`
- `16MB flash`
- `8MB embedded PSRAM`

То есть железо оказалось нормальным, а проблема была в описании платы внутри проекта.

---

## <span style="color:#4EA1FF;">Что именно исправило ситуацию</span>

Было сделано три обязательных шага.

### <span style="color:#67C587;">1. Собственный board profile</span>

Мы создали локальный профиль:

- [boards/matrixclock_esp32s3_n16r8.json](./boards/matrixclock_esp32s3_n16r8.json)

Он задаёт ключевые параметры платы:

- `flash_size = 16MB`
- `maximum_size = 16777216`
- `PSRAM`
- правильные build flags для `ESP32-S3 N16R8`
- корректный класс модуля для нашего проекта

### <span style="color:#67C587;">2. Явное использование этого профиля в PlatformIO</span>

В [platformio.ini](./platformio.ini) проект переключён на:

- `board = matrixclock_esp32s3_n16r8`
- `boards_dir = boards`

То есть PlatformIO теперь берёт описание платы не из общего шаблона, а из нашего локального профиля.

### <span style="color:#67C587;">3. Собственная partition table под 16MB</span>

Проект использует:

- [partitions_ota_16mb.csv](./partitions_ota_16mb.csv)

Это важно, потому что одной строки `flash_size = 16MB` недостаточно.
Нужно ещё реально разложить разделы под нужный объём флеша.

---

## <span style="color:#FF9B5E;">Особенно важный подводный камень</span>

Для `LittleFS` в Arduino-ESP32 раздел должен называться именно:

```csv
spiffs, data, spiffs, ...
```

А не, например:

```csv
littlefs, data, spiffs, ...
```

Это был отдельный критичный баг:

- файловая система физически прошивалась
- но `LittleFS.begin()` не находил раздел
- вебморда не поднималась
- устройство писало, что `index.html missing`

То есть при использовании `LittleFS` имя раздела в CSV должно всё равно быть `spiffs` в первом столбце.
Это особенность совместимости Arduino-ESP32, а не логическая красота названий.

---

## <span style="color:#4EA1FF;">Почему VS Code / PlatformIO показывали не то</span>

Нужно понимать важную вещь:

- VS Code сам по себе ничего “не знает” о железе
- он показывает то, что сообщает PlatformIO
- PlatformIO опирается на board profile и project config

Если профиль платы описан как `N8`-подобный вариант, то IDE будет вести себя так, будто это и есть реальность.

Поэтому когда IDE показывала ограничения на `8MB`, это было следствием неправильного профиля, а не доказательством того, что модуль фальшивый.

---

## <span style="color:#67C587;">Как правильно создавать профиль для такой платы</span>

Если нужно поднять похожий модуль с нестандартной памятью, рабочий порядок такой:

1. Подтвердить параметры чипа не по наклейке, а инструментом чтения flash/psram.
2. Не доверять вслепую стандартному board profile.
3. Создать локальный JSON-профиль в `boards/`.
4. Явно указать:
   - `flash_size`
   - `maximum_size`
   - `PSRAM`
   - build flags под нужный класс модуля
5. Подключить этот профиль через `boards_dir` и `board = ...`.
6. Создать собственную partition table под нужный объём флеша.
7. Проверить, что раздел для `LittleFS` назван `spiffs`.
8. После смены partition table обязательно сделать хотя бы один USB-проход, потому что OTA не переучит разметку флеша магически.
